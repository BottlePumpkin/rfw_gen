# Dogfood Skill Design Spec

## Overview

`/dogfood` 스킬 — rfw_gen 패키지를 "처음 쓰는 사용자" 페르소나로 직접 사용해보며 이슈를 발견하고, 패키지를 지속적으로 개선하는 자동 피드백 루프.

## 동기

실제 사용자의 관점에서 rfw_gen, rfw_gen_builder, rfw_gen_mcp, rfw_preview를 사용해봐야 DX 문제, 문서 부족, 기능 갭 등을 발견할 수 있다. 이를 Claude가 자동으로 수행하여 무한히 반복 가능한 피드백 루프를 구축한다.

## 전제 조건

- `.gitignore`에 `example2/` 항목이 존재해야 함 (첫 실행 시 스킬이 자동 추가)
- `gh` CLI 인증 완료 상태
- GitHub 레포에 `dogfood`, `bug`, `dx`, `feature`, `docs` 라벨 존재 (첫 실행 시 스킬이 누락 라벨 자동 생성)

## 모드

### `/dogfood run` — 수집 사이클

**흐름:**

1. 전제 조건 확인 (`.gitignore`, GitHub 라벨)
2. 이전 `example2/` 디렉토리 존재 확인 → 있으면 삭제
3. 페르소나 랜덤 선택 (3종 중 1개)
4. 앱 주제 랜덤 선택 (이전 사이클과 겹치지 않게)
5. `flutter create example2` + pub.dev 의존성 설정
6. 페르소나 관점으로 앱 개발 시도
   - 해당 페르소나 수준에 맞는 정보만 참고
   - 막히는 지점, 에러, 혼란스러운 부분 모두 기록
   - 최소 5-8개 이상의 서로 다른 RFW 위젯을 사용하며, 레이아웃과 인터랙션 패턴을 모두 포함
7. 이슈 목록 정리 (이슈 + 카테고리 + 해결방안, 최대 7건)
8. 중복 검사: 기존 `dogfood` 라벨 이슈와 제목/키워드 비교, 중복이면 스킵
9. GitHub Issue 개별 등록 (`dogfood` 라벨)
10. 사이클 요약 출력

**이슈가 0건인 경우:** "Clean cycle" 요약을 출력하고 사이클 번호는 정상 증가시킨다.

### `/dogfood fix` — 수정 세션

**흐름:**

1. GitHub에서 `dogfood` 라벨 열린 Issue 목록 조회
2. 이슈를 우선순위별로 정렬하여 제시 (`bug` > `dx` > `feature` > `docs`)
3. 사용자가 수정할 이슈 선택 (또는 Claude 추천 수락)
4. `git-workflow` 스킬을 통해 worktree 생성: `/git-workflow start fix/dogfood-{issue번호}`
5. 패키지 개선 → 테스트 → PR 생성
6. 완료된 Issue에 PR 링크 코멘트

## 페르소나 시스템

| 페르소나 | 특성 | 발견하는 이슈 유형 |
|---------|------|------------------|
| **신규 사용자** | Flutter 숙련, rfw_gen 처음. README와 pub.dev만 보고 시작 | 온보딩 마찰, 문서 부족, "첫 5분" 경험 |
| **문서 읽은 사용자** | Flutter 숙련, rfw_gen 문서 좀 읽어봄. 기본 개념은 이해 | API 직관성, 중급 사용 패턴, 에러 메시지 품질 |
| **코드젠 경험자** | build_runner, freezed 등 경험 있음. 코드젠 패턴에 익숙 | 컨벤션 불일치, 고급 기능 부재, 다른 패키지 대비 갭 |

**행동 규칙:**
- 각 페르소나는 자기 수준에 맞는 정보만 참고
- 막히면 실제 사용자처럼 "시도 → 실패 → 다른 방법 시도" 과정을 거침
- 코드젠 경험자는 freezed, json_serializable 등과 비교하며 평가

**페르소나별 참조 가능 범위:**

| 페르소나 | 참조 가능 자료 | 참조 불가 |
|---------|-------------|----------|
| **신규 사용자** | README.md, pubspec.yaml, pub.dev 페이지 | 소스코드, 내부 문서, 생성된 파일 |
| **문서 읽은 사용자** | 위 + CHANGELOG, doc/, example/ 코드 | 패키지 소스코드 내부 |
| **코드젠 경험자** | 모든 파일 (소스코드, build.yaml, 생성된 파일 포함) | — |

## 앱 주제

매 사이클마다 Claude가 랜덤으로 선택. 예시:
- 투두 앱, 날씨 대시보드, SNS 피드, 레시피 북
- 운동 트래커, 음악 플레이어 UI, 설정 화면
- 뉴스 리더, 채팅 UI, 이커머스 상품 목록

각 주제는 최소 5-8개의 서로 다른 RFW 위젯을 사용하고, 레이아웃과 인터랙션 패턴을 모두 포함해야 한다. 기존 dogfood Issue의 사이클 주제를 확인하여 중복 방지.

## 이슈 카테고리

| 라벨 | 설명 | 우선순위 |
|-----|------|---------|
| `bug` | 빌드 에러, 코드젠 실패, 잘못된 출력 | 1 (최고) |
| `dx` | 문서 부족, API 직관성, 에러 메시지 품질 | 2 |
| `feature` | 미지원 위젯, 부족한 기능 | 3 |
| `docs` | 문서 개선, 예시 부족 | 4 |

**사이클당 최대 7건.** 8건 이상 발견 시 우선순위로 상위 7건만 등록하고, 나머지는 사이클 요약에 "다음 사이클로 이연" 으로 기록.

## GitHub Issue 템플릿

```markdown
Title: [dogfood/{카테고리}] {이슈 요약}

## 발견 컨텍스트
- 사이클: #{번호}
- 주제: {앱 주제}
- 페르소나: {페르소나 이름}

## 문제
{무엇을 하려다 어떤 문제를 겪었는지}

## 재현 방법
{구체적 단계}

## 제안하는 해결 방안
{어떻게 개선하면 좋을지}

## 영향받는 패키지
- [ ] rfw_gen
- [ ] rfw_gen_builder
- [ ] rfw_gen_mcp
- [ ] rfw_preview
```

**라벨:** `dogfood` (공통) + 카테고리 라벨 (`bug`, `dx`, `feature`, `docs`)

## 사이클 요약 출력

```
## Dogfood Cycle #N Summary

📋 주제: {앱 주제}
🎭 페르소나: {페르소나 이름}
📦 의존성: rfw_gen ^x.y.z, rfw_preview ^x.y.z

### 발견된 이슈 (N건)
| # | 카테고리 | 이슈 | GitHub |
|---|---------|------|--------|
| 1 | bug     | ...  | #nn    |
| 2 | dx      | ...  | #nn    |

### 다음 사이클로 이연 (M건)
- ...
### 중복으로 스킵 (K건)
- ...
```

## 상태 관리

- GitHub Issue가 유일한 진실의 원천
- 사이클 번호: `dogfood` 라벨 Issue 검색 → 제목/본문에서 `사이클: #N` 패턴 파싱 → 최대값 + 1
- 주제 중복 방지: 기존 Issue의 사이클 주제를 확인하여 겹치지 않게 선택
- 이슈 중복 방지: 새 이슈 등록 전 기존 열린 `dogfood` Issue 제목과 키워드 비교

## example2 프로젝트 설정

- 위치: 프로젝트 루트의 `example2/`
- 의존성: pub.dev 배포 버전 참조 (path가 아닌 버전 지정)
- 라이프사이클: 다음 `/dogfood run` 실행 시 삭제 후 재생성
- `.gitignore`에 `example2/` 추가하여 커밋 대상에서 제외

## 에러 처리

| 상황 | 대응 |
|-----|------|
| `flutter create` 실패 | 에러 메시지 출력 후 사이클 중단 |
| `pub get` 실패 | 에러 메시지 출력 후 사이클 중단 (pub.dev 버전 문제 가능성 안내) |
| `gh issue create` 실패 | 1회 재시도 → 실패 시 이슈 목록을 터미널에 출력하고 수동 등록 안내 |
| 이슈 0건 발견 | "Clean cycle" 요약 출력, 사이클 번호 정상 증가 |
| `gh` 인증 안 됨 | 사이클 시작 전 감지하여 `gh auth login` 안내 후 중단 |

## 기존 스킬/에이전트 연계

- `/dogfood fix` 실행 시 `git-workflow` 스킬의 worktree 패턴 활용 (`/git-workflow start` 호출)
- 이슈 수정 시 기존 테스트 인프라 (golden test 등) 활용
